## Architecture

### Stack
- **Frontend:** React 18, TypeScript, Vite (SWC), React Router, Tailwind CSS v4
- **UI Components:** shadcn/ui (Radix UI primitives) in `src/components/ui/`
- **Backend:** Supabase Edge Functions (Deno) + Hono web framework
- **Database:** Supabase KV Store — flat key/value with JSON blobs (not relational)
- **State:** React hooks + Context API only. No Redux/Zustand.
- **Auth:** Mock login stored in `localStorage`. No real Supabase Auth.

### Key Directories
- `src/components/` — All UI components. Subdirectories: `operations/`, `accounting/`, `reports/`, `hr/`, `admin/`, `shared/`, `selectors/`, `ui/`, `design-system/`
- `src/hooks/useUser.tsx` — User context and auth state (`useUser()` hook)
- `src/types/` — Shared TypeScript type definitions
- `src/styles/globals.css` — Design tokens (CSS vars), Tailwind v4 `@source` config
- `supabase/functions/server/index.tsx` — Entire Hono server (~9,400 lines)
- `supabase/functions/server/kv_store.tsx` — KV helper functions
- `supabase/functions/server/seed_data.tsx` — Demo data

### Routing (App.tsx)
Routes are flat and module-scoped:
- `/dashboard`, `/activity-log`, `/profile`, `/admin`
- `/operations/import-bookings`, `/export-bookings`, `/trucking`, `/others`, `/clients`
- `/accounting/vouchers`, `/billings`, `/collections`, `/expenses`
- `/reports/*`, `/hr`

### Data Flow
1. Component → `fetch()` → Hono server endpoint
2. Hono handler → Supabase KV Store (key prefixes: `booking:`, `billing:`, `collection:`, `expense:`, `voucher:`, `client:`, `user:`, etc.)
3. JSON response → component local state

KV store is denormalized — relationships are ID references embedded in JSON blobs, not relational joins.

### Business Logic Flow
```
Operations (booking created)
  → Trucking (optional, linked via linkedBookingId)
  → Vouchers (supplier authorization)
  → Expenses (cost line items)
  → Billings (client invoice)
  → Collections (payment with multi-billing allocation)
```

Key rules:
- Import bookings have 8 pre-populated expense particulars; Export has 6 categories
- Export supports multi-currency (PHP/USD/RMB) with exchange rate conversion
- Collections use allocation model — one payment can cover multiple billings
- Outstanding balance = `totalAmount − sum(allocations)`, computed server-side

### Design System

- Primary green: `#237F66` (brand), `#1E6D59` (hover), `#E8F2EE` (light bg)
- Text: `#12332B` (primary), `#6B7A76` (muted)
- Cards: white bg, `#E5ECE9` border, 10px radius — no shadows (stroke-first)
- Base spacing unit: 4px
- Financial data must use tabular numerals (`font-variant-numeric: tabular-nums`)

## Karpathy Skills — Coding Discipline Principles

Behavioral guidelines to reduce common LLM coding mistakes. Bias toward caution over speed; for trivial tasks, use judgment.

### 1. Think Before Coding

**Don't assume. Don't hide confusion. Surface tradeoffs.**

Before implementing:
- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them — don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, stop. Name what's confusing. Ask.

### 2. Simplicity First

**Minimum code that solves the problem. Nothing speculative.**

- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" or "configurability" that wasn't requested.
- No error handling for impossible scenarios.
- If you write 200 lines and it could be 50, rewrite it.

Ask yourself: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

### 3. Surgical Changes

**Touch only what you must. Clean up only your own mess.**

When editing existing code:
- Don't "improve" adjacent code, comments, or formatting.
- Don't refactor things that aren't broken.
- Match existing style, even if you'd do it differently.
- If you notice unrelated dead code, mention it — don't delete it.

When your changes create orphans:
- Remove imports/variables/functions that YOUR changes made unused.
- Don't remove pre-existing dead code unless asked.

The test: Every changed line should trace directly to the user's request.

### 4. Goal-Driven Execution

**Define success criteria. Loop until verified.**

Transform tasks into verifiable goals:
- "Add validation" → "Write tests for invalid inputs, then make them pass"
- "Fix the bug" → "Write a test that reproduces it, then make it pass"
- "Refactor X" → "Ensure tests pass before and after"

For multi-step tasks, state a brief plan:
```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

**These guidelines are working if:** fewer unnecessary changes in diffs, fewer rewrites due to overcomplication, and clarifying questions come before implementation rather than after mistakes.
